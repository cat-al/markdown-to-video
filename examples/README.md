## `examples` 目录说明

这里存放所有 Markdown 文稿样例，按用途分两类：

- `examples/demo/`：演示稿、实验稿、功能验证样例
- `examples/published/`：已经发布或接近发布状态的成型文稿

> 除了本文档，写稿前请同时参考：
>
> - `docs/markdown-guide.md`：Markdown 语法、frontmatter、控制指令、布局清单
> - `docs/文案生成规则.md`：精读 / 解读类中文视频的开头、标题、语气规则

---

### 目录分工

#### `demo/`

适合放：

- 默认演示案例
- TTS / 布局 / 流程验证稿
- 还在试结构或试风格的草稿

#### `published/`

适合放：

- 已经发布的视频文案
- 已经成型、可直接复用的精读稿
- 后续只做小修而不是大改结构的成稿

命名规则：

- 使用三位编号前缀：`001-xxx.md`
- 新稿按发布时间或入库顺序递增编号
- 已经对外发布后，尽量不要频繁改编号

#### 放置规则

- 还在试格式、试节奏、试配音 → 放 `demo/`
- 已经发布或准备发布 → 放 `published/`
- 稿子从试验稿变正式稿，再从 `demo/` 移到 `published/`

### Markdown 写作强约束

下面这些是写稿过程中**必须遵守**的写作约束，否则生成的视频会出问题。
写完初稿后建议对照 [自检清单](#写稿自检清单) 过一遍。

#### 一、分页与旁白

- 使用 `---` 单独成行作为分页符
- 每页一个明确的观点，页与页之间有推进关系
- 每页必须写 `<!-- voiceover ... -->` 作为该页配音文本，**不要只靠正文靠 TTS 生成**
- 旁白要更口语，但必须覆盖正文核心信息，不允许脱离页面内容另讲一套

#### 二、正文长度

- 每页 bullet 控制在 **3 到 5 条**，超过 6 条会超出大多数布局的展示容量
- 正文段落尽量简短，每段一两句话，避免长段落
- 不要把多个观点塞在同一个 bullet 里

#### 三、代码块

**代码块长度限制在 15 行以内**（包含空行）。原因：

- 超过 15 行会在 `code` 布局里变得很小、看不清
- 代码块是用来示意写法的，不是粘贴整个文件的
- 长代码里真正关键的部分会被淹没

如果原始代码很长，做下面几件事之一：

- 只保留最能体现观点的核心片段，其他用注释 `// ...` 省略
- 拆成两页代码，每页讲一个点
- 不用代码，用 bullet 说清楚就行

#### 四、中英文与符号

- Markdown 中的专有名词按正确写法大小写：`Claude.md` 而不是 `CLAUDE.md`，`GitHub` 而不是 `github`
- 中文正文之间的英文词、数字前后加空格：`使用 Go 1.22`、`控制在 50 到 80 行`
- 不要使用全角符号替代半角符号（逗号、冒号除外，按中文习惯）

#### 五、布局选择

布局控制策略（详见 `docs/markdown-guide.md`）：

- **普通内容页**：不写 `<!-- layout: xxx -->`，让系统自动选型
- **关键页**：手动覆写布局，例如开场、章节过渡、金句总结、结尾提醒
- **布局决策留在 Markdown 内**，不要为了单篇文章去改视频组件源码

下面是最常踩坑的几种情况：

| 症状 | 原因 | 推荐布局 |
| --- | --- | --- |
| 金句 / 一句话结论没展示 | 被当成普通段落，被布局截断 | `accent-bar` / `centered` / `quote` |
| bullet 文字太长，挤在小圆圈里 | 默认选到了 `radar` | 改 `argument` / `waterfall` / `checklist` |
| 正文两段只显示一段 | 布局只取 `paragraphs[0]` | `split-list` / `sidebar-note` |
| 想分两组（如"约定类 / 命令类"） | 布局只会按索引分列，丢失分组语义 | 用 `compare`，并在每个 bullet 前加`【约定】` `【命令】`前缀 |
| 需要同时显示要点和一条金句结论 | 普通布局不展示 blockquote | `manifesto`（blockquote 变 thesis 卡片） |

#### 六、关键句的展示

**如果某句话必须出现在视频里**（比如"一句话判断"这种金句），它不能只是正文里的一行 `**粗体**`，因为很多布局不会展示它。

必须至少满足以下一种：

- 用 `>` blockquote 包裹，并选用会展示 blockquote 的布局（`manifesto`、`quote`、`spotlight`）
- 拆成独立一页，用 `<!-- layout: accent-bar -->` 或 `<!-- layout: centered -->`
- 放进 `heading` 里（不推荐，heading 不是金句载体）

不要假设布局会自己把所有正文都展示出来。

#### 七、Bullet 分组

Markdown 的 bullet 是扁平列表，**所有 bullet 会被解析成一个平铺数组**，`## 约定类：` / `## 命令类：` 这种小分类标题在渲染时会丢失。

如果想表达分组，做下面之一：

- 把分类名作为前缀放进每个 bullet：`- 【约定】所有 handler 放在 internal/handler/`
- 拆成两页，每页只讲一组
- 用 `compare` 布局，让前一半和后一半自然分列（配合前缀更清晰）

### 写稿自检清单

写完初稿后过一遍：

- [ ] Frontmatter 里 `title` / `subtitle` / `themeColor` 已填
- [ ] 第一页交代了 **精读什么、来自哪里、为什么值得看**（详见 `docs/文案生成规则.md`）
- [ ] 每页有 `<!-- voiceover ... -->` 旁白
- [ ] 每页 bullet ≤ 5 条
- [ ] 代码块 ≤ 15 行
- [ ] 专有名词大小写正确（`Claude.md` 等）
- [ ] 想展示的金句单独成段或独立成页
- [ ] 有分类语义的 bullet 加了前缀或拆成多页
- [ ] 最后一页有一句话总结
