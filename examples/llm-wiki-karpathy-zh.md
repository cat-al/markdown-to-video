# LLM Wiki：它想把知识管理变成什么样？

今天我们精读下 Andrej Karpathy 发布在 GitHub Gist 上的文章《LLM Wiki》，看看它为什么主张让大模型持续维护一套会演化的 wiki，而不是每次都从原始资料里临时检索答案。

Karpathy 是 OpenAI 创始成员之一，也曾担任 Tesla AI 总监，在深度学习和 AI 教育领域影响很大。

- 分享位置：GitHub Gist
- 文章标题：`LLM Wiki`
- 核心提案：让 LLM 持续维护一套会增长、会修订、会互相链接的 wiki
- 目标：把知识从一次次检索，变成长期积累的结构化资产

<!-- voiceover
今天我们精读下 Andrej Karpathy 发布在 GitHub Gist 上的一篇文章，标题叫 LLM Wiki。它最核心的想法，是把知识库从一次次临时检索，变成一套由大模型持续维护、持续更新、持续关联的 wiki。也就是说，重点不只是这是谁写的，而是 LLM Wiki 这套思路到底想解决什么问题：让知识能够真正沉淀、复用，并随着新资料不断生长。

## 它和传统 RAG 的关键差别

在传统 RAG 里，模型几乎每次都要“重新发现知识”。

- 每个问题都重新检索片段
- 多文档综合时，重复拼接成本很高
- 结论不会沉淀成长期资产
- 细微矛盾也很难被持续追踪

**LLM Wiki 的核心差别是：wiki 本身就是可持续演化的中间层。**

<!-- voiceover
Karpathy 认为，传统 RAG 最大的问题是没有积累。每来一个问题，模型都要重新检索、重新理解、重新组合。尤其当一个问题需要跨越五篇甚至十篇资料时，这种重复劳动会越来越明显。而 LLM Wiki 的关键差别，是把 wiki 变成一个长期存在的中间层。知识不是每次重新推导，而是先被整理进一个持续演化的结构化系统里。
-->

---

## 这个系统有三层结构

1. **Raw sources**：原始资料，只读、不改，是事实来源
2. **Wiki**：由 LLM 维护的 markdown 页面，负责总结、关联、修订
3. **Schema**：告诉 LLM 应该怎样组织 wiki、怎样工作、怎样维护规范

这一层 `schema` 很关键，它决定 LLM 是随便聊天，还是像一个有纪律的知识库维护者。

<!-- voiceover
整套架构其实很清晰，可以理解成三层。第一层是原始资料，也就是文章、论文、图片、数据文件，这一层只读，不应该被模型修改。第二层是 wiki，本质上是一组由 LLM 维护的 markdown 页面，负责把知识提炼成摘要、实体页、概念页和综合页。第三层是 schema，也就是规则文档，用来规定这套 wiki 的组织方式和工作流。Karpathy 特别强调，真正让系统稳定运转的，往往不是模型本身，而是这一层 schema。
-->

---

## 三个核心操作：Ingest、Query、Lint

- **Ingest**：读入新资料，生成摘要，更新相关页面，补充索引和日志
- **Query**：基于 wiki 回答问题，并把高价值答案反写回 wiki
- **Lint**：定期检查冲突、陈旧结论、孤儿页面和缺失链接

这意味着知识库不是静态仓库，而是一个**不断被维护的工作系统**。

<!-- voiceover
文章把这套系统的日常运转，概括成三个操作。第一个是 ingest，也就是摄入新资料。模型不仅要读资料，还要生成摘要，更新相关页面，并同步修改索引和日志。第二个是 query，也就是基于现有 wiki 回答问题，而且高价值答案还可以反写回 wiki，成为新的知识页面。第三个是 lint，也就是健康检查，用来发现矛盾、过时内容、孤儿页面和缺失引用。这样一来，知识库就不再是一个静态仓库，而是一套不断被维护的工作系统。
-->

## 为什么 `index.md` 和 `log.md` 很重要

这两个文件分别解决“wiki 里现在有什么”和“最近又改了什么”。

- `index.md`：像目录地图，告诉模型有哪些页面、各自讲什么、彼此怎么分类
- `log.md`：像变更记录，告诉模型最近一次 ingest、query、lint 到底改了哪里
- 二者配合后，模型不用每次全盘重读，就能先掌握结构，再追踪最新变化

```md
# index.md
- concepts/llm-wiki.md：LLM Wiki 总览，介绍三层结构和工作流
- concepts/rag.md：传统 RAG 的局限，以及它和 LLM Wiki 的差别

# log.md
## [2026-04-02] ingest | Karpathy - LLM Wiki
- 新增 `sources/karpathy-llm-wiki-summary.md`
- 更新 `concepts/llm-wiki.md`
- 在 `concepts/rag.md` 补充“与 RAG 的差别”
```

模型先读 `index.md` 知道该去哪里找，再读 `log.md` 知道最近发生了什么，然后只打开真正相关的页面。

<!-- voiceover
Karpathy 专门提出了两个很实用的文件。index 的作用，是帮助模型先建立全局地图，知道这套 wiki 里有哪些页面、每页大概讲什么。log 的作用，是记录最近的变更，让模型快速理解这次到底新增了什么、改动了什么、修订了什么。所以这段代码并不是在展示某种复杂语法，而是在展示一种非常实用的工作习惯：先用 index 找路，再用 log 看变化，然后只进入真正相关的页面。这个设计很朴素，但对可维护性非常重要。
-->

## 查询的结果，也应该继续沉淀

LLM Wiki 不只是在回答问题，它还在**制造新的知识页面**。

- 一次对比分析，可以存成比较页
- 一个跨主题洞察，可以存成综合页
- 一份演示文稿，可以由 wiki 内容直接生成
- 聊天里产生的好结论，不该只留在对话历史里

<!-- voiceover
文章里一个很有启发性的点是，查询的结果本身也应该继续沉淀。比如你让模型做一次对比分析，这份分析不应该随着聊天结束而消失，而应该回写成新的 wiki 页面。你发现了一个跨主题联系，也应该把它沉淀成新的综合结论。换句话说，查询不是消费知识，而是在继续生产知识。
-->

## 它为什么适合很多场景

Karpathy 列了不少应用方向，本质上都满足同一个条件：**知识会持续累积，而且需要被组织起来。**

- 个人成长与自我追踪
- 研究主题的长期深入
- 读书过程中的人物、主题、线索整理
- 团队内部知识库与项目纪要
- 竞品分析、课程笔记、旅行规划、兴趣研究

<!-- voiceover
这套模式并不只适合研究论文。Karpathy 认为，只要一个场景里知识会不断累积，而且你希望它们被系统化整理，这个模式就成立。它可以用于个人成长记录，也可以用于长期研究、读书笔记、团队知识库、竞品分析、课程学习，甚至旅行规划。关键不在领域，而在于你是否需要一个会随着时间不断变强的知识结构。
-->

## 为什么这件事适合交给 LLM

维护知识库最耗人的，通常不是“阅读”本身，而是这些琐碎工作：

- 更新交叉引用
- 修订旧摘要
- 标记新旧观点的冲突
- 保持几十个页面之间的一致性

**人负责判断什么重要，LLM 负责把维护成本降到接近零。**

<!-- voiceover
Karpathy 解释这套方法为什么现实可行时，强调了一个常被忽略的事实：人类放弃维护 wiki，往往不是因为不愿意思考，而是因为 bookkeeping 太烦了。你要补链接、改摘要、处理矛盾、保持几十个页面的一致性，这些事情价值很高，却特别消耗耐心。而 LLM 恰好适合承担这种大规模、重复、细碎的维护工作。于是人类可以把注意力集中在选资料、提问题和做判断上。
-->

## 文章最后的提醒

这不是一套固定产品方案，而是一种模式。

- 目录结构可以自定义
- 页面格式可以自定义
- 工具链可以简单，也可以逐步增强
- 小规模时甚至不需要向量检索和复杂基础设施

**这篇文章真正交付的，不是实现细节，而是一种“如何让知识持续增值”的工作框架。**

<!-- duration: 8 -->

文章最后特意提醒，这不是一个标准化产品说明书，而是一种模式。你的目录结构、页面格式、工具选型，甚至是否需要搜索引擎，都可以按自己的场景决定。它真正提供的，不是一套固定实现，而是一个高价值框架：不要让知识每次都从零开始被检索，而要让它被持续整理、持续修订、持续复用。换句话说，知识管理的重点，不再只是找到答案，而是构建一个会不断增值的知识系统。

## 一句话总结

从这篇文章里，可以提炼出一个非常强的判断：

> **RAG 更像即时问答系统；LLM Wiki 更像持续编译中的个人知识操作系统。**

如果说原始资料是代码，schema 是规范，那么由 LLM 持续维护的 wiki，就是那个真正不断变强的“运行中产物”。
