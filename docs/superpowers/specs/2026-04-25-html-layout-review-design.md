# HTML Layout Review 设计文档

**日期**: 2026-04-25
**状态**: 已批准
**适用范围**: `html-layout-review`、`markdown-to-html`

## 1. 概述

在现有 `markdown-scriptwriter -> markdown-to-html` 管线中，补充一个新的独立子 skill：`html-layout-review`。

它的职责不是生成页面，也不是修改页面，而是对已经生成完成的 HTML 幻灯片页面进行**固定画布视觉验收**，重点发现以下会影响最终录屏或交付质量的问题：

- 页面样式错位
- 字体或卡片超出边界
- 文本被截断或裁切
- 元素互相遮挡
- 关键内容贴边过近
- 单页信息密度失衡

该 skill 需要保持为**独立可调用**的检查步骤，但 `markdown-to-html` 在完成 HTML 生成后，必须**提醒用户执行一次该审查 skill**。

## 2. 设计目标

本次新增 `html-layout-review` 的目标是：

1. 为 `markdown-to-html` 生成结果增加一个专门的验收环节
2. 以最终渲染结果为准，而不是以源码表面结构为准
3. 在固定 `1920x1080` 画布下检查 HTML 幻灯片是否可安全录屏或交付
4. 只输出结构化问题清单，不在该阶段直接修复 HTML / CSS
5. 保持项目现有的 skill 组织方式——**平铺在仓库根目录**，而不是迁移到 `.codebuddy/skills/`

## 3. 目录结构

新增目录采用与现有 `markdown-scriptwriter`、`markdown-to-html` 一致的平铺方式：

```text
markdown-to-video2/
├── html-layout-review/
│   ├── SKILL.md
│   └── references/
│       └── review-checklist.md
├── markdown-to-html/
│   └── SKILL.md
└── docs/
    └── superpowers/
        └── specs/
            └── 2026-04-25-html-layout-review-design.md
```

设计原则：

- `SKILL.md` 只保留最核心的职责、触发条件、执行流程和输出约束
- 详细检查项沉淀到 `references/review-checklist.md`
- 首版**不强制引入 `scripts/`**，避免为“只出问题清单”的场景过早增加复杂度

## 4. skill 命名与元信息风格

### 4.1 skill 名称

最终名称定为：`html-layout-review`

选择该名称的原因：

- 直接覆盖目标对象：`html`
- 明确聚焦布局与可视呈现问题：`layout`
- 明确是检查 / 审查行为：`review`
- 触发词容易命中用户描述中的“检查页面”“看有没有溢出”“有没有截断”等需求

### 4.2 frontmatter 风格

`SKILL.md` 的 frontmatter 采用简洁风格，只保留：

- `name`
- `description`

`description` 应只描述**何时触发**，不在 frontmatter 中概述完整流程。

推荐方向：

```yaml
---
name: html-layout-review
description: Use when reviewing generated HTML slides or presentation pages for layout issues such as overflow, clipping, text truncation, element overlap, or content leaving the 1920x1080 frame.
---
```

该写法吸收 `skill-creator` 的内部风格：

- 元信息简洁
- 以触发条件为主
- 覆盖关键症状词：`overflow`、`clipping`、`text truncation`、`element overlap`、`1920x1080 frame`

## 5. 职责边界

### 5.1 负责的事情

`html-layout-review` 负责：

- 审查已经生成好的 HTML 幻灯片页面
- 以最终渲染结果为准，检查页面是否存在影响交付的视觉问题
- 在固定 `1920x1080` 审查基准下逐页检查
- 输出结构化问题清单

### 5.2 明确不负责的事情

`html-layout-review` 不负责：

- 生成 HTML 页面
- 自动修复 HTML / CSS
- 给出泛泛的“审美评价”
- 代替 `frontend-design` 做设计风格判断
- 检查业务内容或文案事实是否正确

一句话定义：

> `html-layout-review` 是一个面向最终渲染结果的 HTML 视觉验收 skill，只报告问题，不直接改页面。

## 6. 审查基准

该 skill 的所有判断必须建立在以下固定基准上：

- **目标画布**：`1920x1080`
- **目标对象**：最终 HTML 渲染结果
- **目标场景**：录屏、视频输出、演示页面交付

关键要求：

- 不以“代码结构看起来合理”替代“页面看起来没有问题”
- 不以浏览器任意缩放后的预览结果作为判断依据
- 重点关注录屏或导出时实际会暴露的问题

## 7. 核心检查项

首版审查聚焦以下几类问题：

### 7.1 错位与重叠

- 模块位置异常偏移
- 标题、正文、图表、卡片之间发生重叠
- 装饰层压住正文层
- 同一页内视觉层次失衡导致阅读路径断裂

### 7.2 越界与裁切

- 标题、正文、数字卡片、图表、装饰元素超出画布边界
- 内容超出自身容器边界
- 元素出现在页面可见区之外

### 7.3 文本截断与溢出

- 字体被裁切
- 长标题换行失控
- 段落或标签内容显示不完整
- 行高、容器高度不足造成文字缺笔画或尾部被切断

### 7.4 安全边距不足

- 关键信息过度贴边
- 多个模块离边缘过近，导致录屏视觉不稳
- 核心内容处于“看似没出界但已经不舒适”的危险区域

### 7.5 信息密度失衡

- 单页内容过多，肉眼可见拥挤
- 模块数量、字号、间距组合不合理，虽未严格越界但已明显影响可读性

## 8. 执行流程

`html-layout-review` 的执行流程应固定为以下五步：

1. **确认审查对象**
   - 找到刚生成的 HTML 页面
   - 明确当前目标是最终交付页面，而不是中间模板文件

2. **固定审查视口**
   - 以 `1920x1080` 为唯一审查基准
   - 必要时强调按实际录屏视角观察页面

3. **逐页检查**
   - 按场景或 slide 逐页审查
   - 优先检查标题区、主体区、边距、安全区、装饰层、图表区域

4. **只记录可见问题**
   - 不输出泛泛设计感受
   - 不输出“也许可以更高级”之类非验收项
   - 只记录会影响交付质量的具体问题

5. **输出结构化问题清单**
   - 汇总结论
   - 给出场景定位、问题类型、描述与严重级别

## 9. 输出格式

该 skill 的输出必须是**问题清单**，而不是修复方案。

推荐结构：

- **审查结论**：通过 / 存在问题
- **问题列表**：
  - 场景编号或页面编号
  - 问题类型
  - 问题描述
  - 严重级别

推荐示例：

- **场景 2 / major**：主标题换成两行后压到了右侧图表，标题块与图表发生重叠
- **场景 4 / critical**：底部数字卡片超出画布下边界，录屏时会被裁掉
- **场景 6 / minor**：左侧说明文字距离边缘过近，安全边距不足

## 10. 问题分级标准

为保证输出稳定，问题分级采用三档：

### 10.1 `critical`

已经明显影响最终交付：

- 核心文字被截断
- 元素超出画布边界
- 关键内容被遮挡
- 页面主要信息无法正常阅读

### 10.2 `major`

当前页面存在明显版式风险：

- 标题压缩主体区
- 多个模块重叠
- 内容块严重挤压
- 结构虽勉强可读，但录屏后质量明显不稳

### 10.3 `minor`

不一定阻塞交付，但应提醒修整：

- 安全边距偏小
- 局部行距过紧
- 视觉压力过高
- 某些模块位置略显别扭但未形成严重故障

## 11. 与 `markdown-to-html` 的联动

`html-layout-review` 是独立 skill，不作为 `markdown-to-html` 的隐式硬依赖，但二者之间必须建立清晰的串联关系。

### 11.1 联动原则

- `markdown-to-html` 负责**生成页面**
- `html-layout-review` 负责**验收页面**
- 生成与验收分离，但要形成稳定工作流

### 11.2 对 `markdown-to-html/SKILL.md` 的要求

在 `markdown-to-html` 完成 HTML 输出后，需要明确提醒用户执行一次 `html-layout-review`。

提醒内容必须覆盖以下意图：

- 页面是否出现错位
- 文字是否超出边界
- 文本是否被截断
- 元素是否跑出 `1920x1080` 画布

推荐提醒语气：

> HTML 生成完成后，建议立即运行 `html-layout-review`，检查页面是否存在错位、文字越界、裁切或溢出问题。

## 12. 参考文件设计

为了避免 `SKILL.md` 过重，详细检查项放在 `references/review-checklist.md`。

该参考文件建议包含：

- 逐页审查清单
- 常见错位类型
- 常见文字截断信号
- 画布安全边距检查点
- 不同页面类型的高风险区域（封面页、数据页、对比页、金句页等）

这样做的目的：

- 保持主 skill 易读
- 让未来扩充检查规则时不需要不断膨胀 `SKILL.md`
- 吸收 `skill-creator` 提倡的“主文件精简、参考材料按需展开”的组织方式

## 13. 验证方式

该 skill 的首版验证可以围绕典型坏页面样例展开，而不要求先写自动化脚本。

建议最少覆盖以下样例：

1. **超长标题页**
   - 检查是否能稳定识别标题换行挤压、超宽或超高问题

2. **高密度信息页**
   - 检查是否能识别卡片互相压迫、文本区过挤、阅读路径混乱

3. **装饰元素较多的页面**
   - 检查是否能识别背景元素遮挡正文、装饰层超出安全区

4. **大字号数字 / 引用页**
   - 检查是否能识别大字被裁切、基线错位、容器高度不足

验收标准：

- 能稳定指出肉眼可见的错位、越界、截断和重叠问题
- 不把输出重点浪费在泛泛审美评论上
- 输出结果具备页面定位和问题分级，方便后续修复环节消费

## 14. 后续扩展空间

本设计首版故意保持收敛，不引入自动修复和复杂脚本。后续可按需要扩展：

- 新增 `scripts/`，做更稳定的浏览器审查辅助
- 增加截图、标注、自动生成问题报告
- 在审查 skill 之后再补一个“修复型 skill”，专门消费问题清单并改 HTML / CSS

但这些都不属于本次设计范围。

## 15. 最终结论

本次设计确认新增一个新的根目录平铺子 skill：`html-layout-review`。

它的定位是：

- 独立可调用
- 面向最终 HTML 渲染结果
- 固定 `1920x1080` 审查基准
- 只输出结构化问题清单，不直接修复页面

同时，`markdown-to-html` 在生成完成后，必须提醒用户执行一次该审查 skill，以把 HTML 生成与页面验收串成稳定工作流。
