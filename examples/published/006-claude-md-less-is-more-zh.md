# 大多数人的 Claude.md 都写多了

今天我们深度解读一下 Claude Code 官方文档中关于 Claude.md 的说明，结合自己的实际使用经验，看看这个文件到底该怎么写，以及为什么大多数人的写法可能从一开始就错了。

- 精读来源：Claude Code 官方文档（code.claude.com/docs）
- 主题：Claude.md 的写法与最佳实践
- 核心问题：这个文件到底该写什么、不该写什么
- 本次重点：为什么"写少"比"写多"更有效

<!-- voiceover
今天我们来聊一个用 Claude Code 的人几乎都会接触到的东西：Claude.md。这个文件官方文档里有说明，社区里也有很多人分享过自己的写法。但我自己用了一段时间之后发现，大多数人的 Claude.md，包括我自己一开始写的，都写多了。今天这期，我们就以官方文档为起点，结合实际踩坑经验，看看这个文件到底应该怎么写。

## 官方文档怎么说的

Claude Code 官方文档里对 Claude.md 的定位很清楚：它是一个项目级的 Markdown 文件，在每次会话开始时自动加载，用来给 Claude 提供持久指令。

- 可以放在项目根目录、用户主目录、甚至组织级路径下
- 支持层级覆盖：项目级 > 用户级 > 组织级
- 子目录里的 Claude.md 不会启动时加载，只在 Claude 读取该目录文件时按需加载
- 官方建议内容包括：构建命令、编码标准、架构决策、常见工作流
- 官方明确说了：**目标控制在 200 行以下，过长会消耗上下文并降低遵守度**

<!-- voiceover
官方文档对 Claude.md 的定位很明确：它是一个自动加载的项目级指令文件。你可以放在项目根目录，也可以放在用户主目录下做全局配置。它支持层级覆盖，子目录还能按需加载。官方建议写的内容包括构建命令、编码标准、架构决策这些。但这里有一句很关键的话，官方原文说的是：目标控制在 200 行以下，过长会消耗更多上下文并降低遵守度。这句话其实已经在暗示一个方向：这个文件，不是越多越好。

## 我一开始也写了一大堆

说实话，我刚开始用 Claude.md 的时候，也是往里猛塞东西。比如下面这种，看起来很完整，但问题也很典型：

```markdown
# Claude.md
## 编码规范
- 错误处理必须用 if err != nil 逐层返回，不要 panic
- 接口命名加 er 后缀，公开函数必须写注释
- 不要使用 init 函数，避免包级别副作用
- 单个函数不要超过 80 行
## 并发规范
- goroutine 必须有明确的退出机制
- channel 优先于 sync.Mutex
- context.Context 作为第一个参数传递
## 测试规范
- 表驱动测试优先
- mock 用 interface 抽象，不要用第三方框架
```

坏就坏在三点：

- **它写的几乎都是语言通识**：这些本来就是 Go 社区的常识，Claude 不看也大概率会这么写
- **它没有提供项目独有信息**：看完这一堆规则，Claude 还是不知道这个项目是干嘛的、目录怎么约定、哪些坑已经踩过
- **它把真正重要的信息淹没了**：整份文件像团队编码规范，却不像写给 AI 的项目入职说明

所以这种写法的问题，不是"内容不对"，而是**内容虽然对，但对这个文件的用途来说几乎没有信息增量**。

<!-- voiceover
我自己一开始也犯了这个错。刚接触 Claude.md 的时候，我写了一份特别详细的规范进去。什么 error handling 逐层返回、接口命名加 er 后缀、不要用 init 函数、goroutine 要有退出机制、测试用表驱动，写了满满一大段。问题是，这些内容看上去都对，但坏就坏在：第一，它们大多是 Go 的通用常识，Claude 本来就知道；第二，它们没有告诉 Claude 任何这个项目独有的信息；第三，它们把真正重要的项目约束全淹没了。结果这份文件读起来像团队编码规范，不像写给 AI 的入职说明。也就是说，它不是写错了，而是写偏了。
-->

## 为什么写多了反而更差

<!-- layout: argument -->

这不只是"没用"的问题，写多了其实有明显的负面效果。

- **占上下文**：Claude.md 会注入每次对话，写越多占位越大，留给真正有用信息的空间就越少
- **稀释关键指令**：50 条规则一起塞进去，重要的 5 条会被淹没，Claude 的遵守度反而下降
- **可能产生冲突**：你写的"通用规范"和模型已有知识如果有微妙差异，反而让 Claude 困惑
- **维护成本高**：写得越多，过时内容越多，最终变成一份没人维护的废文档

官方文档自己也说了：过长会降低遵守度。这不是理论推测，是实际观察。

<!-- voiceover
你可能觉得，写多了顶多没用，不会更差吧。其实不是。Claude.md 的内容会注入到每次对话的上下文里。你写得越多，占的上下文空间就越大，留给真正有用信息的空间就越小。而且当你一次性塞进去 50 条规则，真正关键的那几条就被淹没了，Claude 的遵守度反而会下降。更微妙的是，你写的通用规范可能和模型本身的知识有微妙差异，这反而会让它困惑。官方文档也直接说了，过长会降低遵守度。所以写多了不是没代价的。
-->

## 核心原则：只写 AI 的盲区

<!-- layout: manifesto -->

砍掉那些废话之后，我总结出一条核心原则：

> **Claude.md 不是规范文档，是写给 AI 的"入职须知"——只写它第一天上班不可能猜到的事。**

- 只写 AI 不知道的事：Claude 不看 Claude.md 能不能自己知道？能就不写
- 不教通用常识：Claude 已经知道 Go 的 error handling、interface 设计、goroutine 用法
- 只写项目特有信息：你要告诉它的，是只有在你这个项目里干过活才知道的事

<!-- voiceover
砍掉那些没用的之后，我现在写 Claude.md 只遵循一条原则：只写 AI 的盲区。判断方法也很简单：如果这条信息 Claude 不看 Claude.md 也能知道，那就不用写。Claude 已经知道 Go 的 error handling 怎么写、interface 怎么设计、goroutine 怎么用，你不需要再教它。你需要告诉它的，是只有在你这个项目里干过活才会知道的事情。换句话说，Claude.md 不是规范文档，是写给 AI 的入职须知。
-->

## 该写什么：业务上下文

<!-- layout: split-list -->

最高价值的内容，是一句话说清这个项目是干嘛的。

这种信息 Claude 从代码里可能要推断很久，但你一句话就能说清楚，让它在回答每个问题时都带着正确的全局理解。

- "本项目是一个 API 网关，负责鉴权、限流和请求转发，下游对接 gRPC 微服务"
- "这是一个内部运营后台，用户是运营团队，不是终端用户"

<!-- voiceover
最高优先级的内容，是业务上下文。用一句话告诉 Claude 这个项目是干嘛的。比如，本项目是一个 API 网关，负责鉴权、限流和请求转发。这种信息 Claude 从代码里可能要推断半天，但你一句话就说清楚了。有了这个全局理解，它后续回答所有问题都会更准。

## 该写什么：踩坑记录和否决决策

第二类高价值内容，是你已经否决的方案和踩过的坑。

- "HTTP 框架用 gin，不要建议换 fiber，迁移成本太高"
- "配置管理用 viper，不要用环境变量直接读取"
- "之前试过 gorm 的自动迁移，线上出过事故，现在统一用 golang-migrate 手动管理"

这类信息的价值在于：防止 Claude 反复建议你已经试过并且放弃的方案。没有这些记录，每次新会话它都可能重新建议你换框架，然后你再解释一遍为什么不行。

<!-- voiceover
第二类特别值得写的，是你踩过的坑和已经否决的方案。比如 HTTP 框架用 gin 不要建议换 fiber，配置管理用 viper 不要直接读环境变量。这类信息的核心价值是防止 Claude 反复建议你已经试过并且放弃的方案。如果你不写，每次新会话它都可能重新提议换框架，然后你再解释一遍为什么不行。写一条永久生效，比每次口头纠正高效得多。

## 该写什么：项目特有约定和常用命令

<!-- layout: compare -->

第三类，是你项目里独有的约定，以及那些 Claude 猜不到的命令。

- 【约定】所有 API handler 必须放在 internal/handler/ 下，按业务域分子目录
- 【约定】数据库迁移文件统一放在 migrations/，命名格式为 YYYYMMDD_描述.sql
- 【约定】配置文件只走 configs/ 目录，不允许散落在各个模块里
- 【命令】本地启动：make run
- 【命令】跑全量测试：make test
- 【命令】生成 protobuf：make proto

这些东西不写的话，Claude 只能猜，猜对了算运气，猜错了浪费你的时间。

<!-- voiceover
第三类是项目独有的约定和常用命令。左边这些是项目约定，比如所有 API handler 必须放在某个目录下，数据库迁移文件怎么命名。右边这些是常用命令，比如本地启动用什么、跑测试用什么、生成 protobuf 用什么。这些东西如果不写进去，Claude 只能靠猜。猜对了是运气，猜错了就是你的时间成本。

## 不该写什么

反过来看，以下内容几乎永远不需要出现在 Claude.md 里：

- **语言语法规范**：Claude 已经知道 Go、Python、TypeScript 怎么写
- **通用最佳实践**：error handling、DRY 原则、接口设计这些不用教
- **项目目录结构**：Claude 能自己看文件树，你不需要手动描述
- **长篇解释**：写结论就够了，不需要写推理过程
- **格式化规则**：如果你有 golangci-lint 或 gofmt 配置，Claude 会自己读

## 格式和维护建议

最后分享几条实操建议。

- **不需要漂亮的 Markdown**：紧凑就行，不要为了格式浪费 token
- **控制在 50 到 80 行**：如果超了，大概率是写了太多 AI 本来就知道的东西
- **分层放置**：根目录放全局规则，子目录放局部约定，比如 `scripts/Claude.md` 只写脚本相关的
- **动态维护**：每次 AI 犯错加一条，每隔一段时间清理过时的条目
- **用 `.claude/rules/` 拆分**：官方支持按路径匹配加载规则文件，大项目可以把规则拆成多个文件

把 Claude.md 当成一份活文档，而不是一次性写死的配置文件。

## 一句话总结

如果把今天的内容压缩成一句话：

> **Claude.md 的最佳状态，是你删不掉任何一行。**

它不是越长越好，而是每一行都必须是 AI 不看就不知道的信息。当你发现自己在写 AI 已经会的东西时，停下来，删掉它。你以为是在帮 Claude，其实是在给它加噪音。
