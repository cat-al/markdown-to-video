## 场景2：这不是工具谁更强的问题

**画面描述**: 画面左右分栏对比。左栏标题"Claude Code"蓝色调，右栏标题"全自动 Agent"橙色调。两栏之间的中轴线上，核心问题"谁该留在回路里？"逐字浮现。分栏内容随字幕逐条渐入。

**视觉元素**:

| 维度 | Claude Code | 全自动 Agent |
|------|------------|-------------|
| 核心理念 | 人机协作，人在回路中 | 自主规划、自主执行、自主修复 |
| 人类角色 | 审查、确认、回滚 | 仅提供需求，看最终结果 |
| 决策点 | 每步或每阶段人工确认 | Agent 自主决策全链路 |
| 核心问题 | 效率够不够？ | 质量稳不稳？ |

> 所以今天要聊的，不只是两个工具的功能对比。一个是 Claude Code，代表人机协作路线。一个是全自动路线上的 Hermes Agent。

> 表面看，它们都在做 AI 编程这件事。但它们真正的分歧，不是谁的模型更聪明。

> 而是一个根本性的架构问题。在 AI 编程这条链路里，人类该不该留在回路中？

## 场景3：全自动真正撞上的墙

**画面描述**: 画面中央是一条不断延伸的自动执行链路，用管道流水线的视觉隐喻。管道前段明亮稳定，中段开始出现裂缝和变色，末段破裂。管道上方浮现三个层次的问题标注。整体从左到右动态展开。

**视觉元素**:

~~~flowchart
任务输入 --> LLM 理解需求 --> 拆解子任务 --> 逐步执行
                                              |
                                    [上下文越来越长]
                                              |
                              自注意力稀释 --> 关键约束丢失
                                              |
                                    错误累积 --> 结果偏离
~~~

- **工程层面**: Agent 框架不成熟、工具调用不稳定 — 这些可以修
- **模型层面**: LLM 自注意力会稀释 — 这是结构性缺陷，不能靠工程绕过
- **根本矛盾**: 链路越长，模型越难保持对初始目标的关注

> 很多人以为全自动做不稳，只是因为 Agent 工程还不成熟。工具调用有 bug，框架还不够完善，再迭代几版就好了。

> 但这篇实践真正揭示的问题，不在工程层面。它撞上的墙，是底层 LLM 的一个结构性缺陷。

> 叫做自注意力的稀释。简单说就是：上下文越长，模型对关键信息的关注度越低。这不是 bug，这是 Transformer 架构的固有特性。

## 场景4：上下文变长 ≠ 模型真的记得住

**画面描述**: 画面展示一条水平展开的长卷轴，代表超长上下文。卷轴前段和末段文字清晰明亮，中段文字逐渐模糊透明。卷轴上方悬浮三个放大镜，分别标注三个核心问题。右侧是数据高亮卡片区域。

**视觉元素**:

- 「200K tokens」上下文窗口 — 看起来很大
- 「中间 40%」信息区域 — 注意力最薄弱的地方
- 「3x」衰减率 — 中段信息被遗漏的概率是首尾的 3 倍

| 误解 | 事实 |
|------|------|
| 上下文长 = 记忆力强 | 能看见 ≠ 能关注，窗口大不代表注意力均匀分布 |
| 放进去就能用 | 中段信息经常被跳过，关键约束"淹没"在海量文本中 |
| 摘要能解决问题 | 摘要是有损压缩，细节和隐含约束会在压缩中丢失 |

> 现在很多人有个错觉。觉得上下文窗口越大，模型就越接近持续思考的能力。但这两件事，完全不是一回事。

> 上下文长，只代表它能看见更多的字。不代表它能稳定地盯住其中真正重要的信息。

> 研究表明，中段信息被遗漏的概率是首尾的三倍。尤其在长链路任务里，早期定下的关键约束最容易失焦。你以为模型记得，但它其实只是"看见过"。

## 场景5：错误传播链 — 为什么 Agent 越跑越偏

**画面描述**: 全屏展示一条从左到右的错误传播链，每个节点是一个执行步骤。正常节点为蓝色圆形，出错节点变为红色并放射出影响箭头。箭头连接到后续多个节点，形成扇形扩散。最后汇聚到"结果全面偏离"的终点。整体有脉冲动画效果。

**视觉元素**:

~~~flowchart
步骤1 [正常] --> 步骤2 [正常] --> 步骤3 [小误判 ⚠️]
                                       |
                                  没被发现纠正
                                       |
                 步骤6 [方向偏移] --> 步骤9 [修复错误对象]
                                       |
                              步骤12 [结果全面偏离 💥]
~~~

- **关键洞察**: 问题不是"一次大失误"，而是"一个小误判的指数级放大"
- **对比**: 人类开发者每几步会回头检查方向 → Agent 只会向前冲
- **结论**: 没有阶段性纠偏机制的长链路执行，必然走向失控

> 这就是很多 Agent 失控的真正根源。不是它完全没看见上下文中的关键信息。而是它没有持续关注那些真正重要的约束。

> 一个早期的小误判，在第三步出现，没有被及时发现。到第六步，方向开始偏移。到第九步，它在修复一个错误的对象。

> 到第十二步，整个结果已经面目全非。你看到的不是一次失误，而是一条错误传播链。每一步的偏差都在被下一步放大，最终指数级爆炸。

## 场景6：Claude Code 更稳，不是因为它更保守

**画面描述**: 画面展示两条平行的流水线对比。上方是全自动流水线（无人检查，一路到底），下方是 Claude Code 流水线（每隔几步有一个人形检查节点，节点处有"确认/回滚"的分叉）。两条线的终点结果用颜色对比：上方红色（偏离），下方绿色（达标）。

**视觉元素**:

~~~flowchart
[全自动路线]
需求 --> 规划 --> 执行 --> 执行 --> 执行 --> ... --> 结果（可能偏离）

[Claude Code 路线]
需求 --> 规划 --> [人工确认] --> 执行 --> 执行 --> [人工审查] --> 执行 --> [人工确认] --> 结果（方向可控）
~~~

| 为什么分段审查有效 | 解释 |
|------------------|------|
| 截断错误传播链 | 误判在第3步被发现，不会扩散到第12步 |
| 补偿注意力稀释 | 人类重新确认关键约束，模型不需要"记住所有" |
| 缩短有效上下文 | 每个分段的上下文更短，模型注意力更集中 |

> 所以 Claude Code 这类人机协作模式更稳定。不是因为它更保守，也不是因为它技术不够先进。而是它正视了一个现实。

> 现有的 LLM 还没有可靠的长链路自治能力。人留在回路里，不是在拖慢效率。人是在做模型做不到的两件事：目标保持和阶段纠偏。

> 每次人工审查，本质上都是在截断可能的错误传播链。这不是权宜之计，这是在模型能力边界内的最优架构。

## 场景7：一句更锋利的结论

**画面描述**: 纯色深色背景，三句结论以大字号逐句打出，每句停留 2 秒。最后一句用强调色高亮处理。留白克制，营造余韵。底部最后渐入频道引导。

**视觉元素**:

:::quote
不是 Agent 不努力，是 LLM 还撑不住。
:::

:::quote
人不是临时补丁，是系统的架构组成部分。
:::

> 所以问题从来不是，全自动听起来够不够高级。真正的问题是：现有 LLM 撑不撑得住长链路自治？

> 在自注意力还会稀释的前提下。在错误会沿着链路指数级放大的现实中。全流程自动化，就很难稳定成立。

> 人不是临时补丁。在今天，人仍然是这套系统的架构组成部分。如果你认同这个观点，点个关注，我们下期继续聊。
