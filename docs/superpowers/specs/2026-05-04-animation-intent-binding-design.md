# 字幕段绑定动画指令 — 解决口播与画面语义脱节

**日期**: 2026-05-04
**状态**: 已实现
**范围**: `markdown-scriptwriter` 格式扩展 + `markdown-to-html` 解析与创作流程变更
**前置文档**: `2026-04-27-subtitle-step-alignment-design.md`

## 1. 背景

当前链路中，`markdown-scriptwriter` 产出的 script.md 和 `markdown-to-html` 创作的 HTML 动画是两个独立过程：

```
script.md
    ├── tts-voiceover: 看 > 段落 → 生成口播音频
    └── markdown-to-html: 看 **视觉元素** → 自由编排 step 动画
```

**核心问题**：`markdown-to-html` 在创作 step 动画时，不知道每个 step 对应的口播段在说什么，导致：
- step 1 动画展示的是概念 A，但口播段 1 在讲概念 B
- 动画顺序和口播叙述顺序不一致
- 最终视频画面和声音在**语义上完全脱节**

根本原因：**口播内容和动画内容之间没有显式绑定关系**，对齐全靠运气。

## 2. 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 绑定方式 | 每段 `>` 口播后紧跟 `**动画**:` 指令 | 物理位置紧邻，不可能错位；阅读直观 |
| 指令详细度 | 描述"画面上发生什么变化"，不写 CSS/SVG 细节 | 给 AI 创作空间，但锁定语义方向 |
| 视觉元素保留 | 作为"素材池"仍放在场景顶部 | 提供可渲染的数据内容，动画指令引用它们 |
| `markdown-to-html` 角色 | 从"自由创作者"变为"指令执行者" | 消除语义错位的根因 |
| 解析变更 | `parse-markdown.js` 新增动画指令解析 | 将指令结构化传递给 AI 创作流程 |

## 3. 新格式规范

### 3.1 格式定义

每段 `>` 口播之后，紧跟一行 `**动画**:` 描述该 step 期间画面上应发生的变化：

```markdown
## 场景2：CLAUDE.md — 给 AI 一份项目说明书

**画面描述**: 整体布局、氛围、动效方向...

**视觉元素**:
（素材池：表格、流程图、代码块等）

| 层级 | 位置 | 作用 |
|------|------|------|
| 全局级 | ~/.claude/CLAUDE.md | 个人偏好 |
| 项目级 | 根目录/CLAUDE.md | 技术规范 |
| 目录级 | 子目录/CLAUDE.md | 模块细节 |

> 大多数人打开 Claude Code，上来就直接问问题。结果每次都要重复告诉它：用什么技术栈、代码风格是什么。

**动画**: Claude Code 对话界面入场，展示重复问"用什么框架？"的循环；重复标记用赤陶红闪烁

> 其实你只需要一个文件，就能让它永远记住这些。这个文件叫 Claude.md，放在项目根目录就行。

**动画**: CLAUDE.md 文件卡片从右侧滑入，逐行显示内容；对话界面中的重复问题淡出消失

> 更强的是，它支持三个层级。全局的管你的个人偏好，项目级的管技术规范，目录级的管模块细节。

**动画**: 三层级关系图入场（全局→项目→目录），陶土棕连线串联，逐层点亮为暖金棕

> 配置好之后，Claude Code 每次对话都会自动加载。再也不用重复交代背景了。

**动画**: 对话界面刷新展示智能回复；"自动加载 ✓"标记浮现；整体画面稳定亮起
```

### 3.2 格式规则

| 规则 | 说明 |
|------|------|
| 绑定关系 | 每段 `>` 口播 **必须** 紧跟一行 `**动画**:` |
| 顺序 | `> 口播段` 在前，`**动画**:` 在后，两者之间只有一个空行 |
| 指令内容 | 描述"这一步画面上发生什么变化"——元素入场/退场/移动/高亮/变形 |
| 指令长度 | 一行，30-100 字，简洁具体 |
| 引用素材 | 可引用 `**视觉元素**` 中定义的数据（表格、流程图节点等） |
| 色彩引用 | 使用品牌色语义词（陶土棕、暖金棕、赤陶红等） |

### 3.3 动画指令编写原则

动画指令描述的是**画面变化**，不是实现细节：

| ✅ 好的写法 | ❌ 坏的写法 |
|------------|------------|
| "三层级关系图入场，逐层点亮" | "用 SVG drawPath 画三条线" |
| "对比表格从左侧滑入" | "translateX(-100%) → 0, duration 600ms" |
| "错误节点变为赤陶红，发射影响箭头" | "fill 从 #7A4F2A 变为 #8B2515" |
| "代码块逐行出现，关键行高亮" | "data-enter-at=2000, opacity 0→1" |

**指令的粒度**：一次画面的"有意义变化"。如果动画指令太复杂（超过 100 字），说明这段口播该拆成两段。

### 3.4 解析规则更新

```
场景内容解析顺序：
  1. **画面描述**: ... → scene.description
  2. **视觉元素**: ... → scene.visualElements
  3. 重复 N 次:
     > 口播段 → scene.steps[N].subtitle
     **动画**: ... → scene.steps[N].animationIntent
```

一个 step 的完整定义 = 口播文本 + 动画指令。

## 4. 对完整链路的影响

```
markdown-scriptwriter → script.md（每段口播绑定动画指令）
    ├──→ markdown-to-html
    │      parse-markdown.js：解析 > 段落 + **动画** 指令 → steps[]
    │      AI 创作：严格按 animationIntent 编排 step 动画
    │
    └──→ tts-voiceover（不变，只看 > 段落）
                                    ↓
                          subtitle-timeline（不变）
                                    ↓
                          video-render（不变）
```

## 6. 各文件详细改动

### 6.1 `parse-markdown.js` 解析逻辑

当前字幕段解析后，新增动画指令检测：

```javascript
// 当前：解析 > 段落
const subtitleMatch = line.match(/^>\s*(.+)/);
if (subtitleMatch) {
  // 收集连续 > 行合并为一段...
  scene.subtitles.push(paragraphText);
  
  // 新增：向后查找紧随的 **动画**: 行
  while (idx < lines.length && lines[idx].trim() === '') idx++; // 跳过空行
  const animMatch = lines[idx]?.match(/^\*\*动画\*\*[:：]\s*(.+)/);
  if (animMatch) {
    scene.animationIntents.push(animMatch[1].trim());
    idx++;
  } else {
    scene.animationIntents.push(''); // 缺失时留空（向后兼容）
  }
  continue;
}
```

输出 JSON 中新增字段：

```json
{
  "scenes": [{
    "sceneNumber": 1,
    "title": "...",
    "description": "...",
    "visualElements": [...],
    "subtitles": ["口播段1", "口播段2", "口播段3"],
    "animationIntents": ["动画指令1", "动画指令2", "动画指令3"]
  }]
}
```

### 6.2 `markdown-to-html/SKILL.md` 创作规则

新增强制规则：

```markdown
### 动画指令驱动（强制）

每个场景的 step 编排**必须严格按照 `animationIntents` 指令执行**：

- step N 的动画内容 = `animationIntents[N]` 描述的画面变化
- 不得自由调换 step 顺序
- 不得忽略任何 animationIntent
- 不得添加 animationIntent 中未提及的 step

**创作流程变更**：
1. 阅读当前场景的完整 `animationIntents` 列表
2. 逐条理解"这一步画面上要发生什么"
3. 根据指令创作对应的 SVG/CSS 动画
4. 确保 stepConfig 中 step 数 == animationIntents 数

**禁止**：
- 看完视觉元素后自己决定"怎么分步展示"
- 把多个 animationIntent 合并为一个 step
- 自行添加 animationIntent 中没有的 step
```

### 6.3 `markdown-scriptwriter/SKILL.md` 格式规范

在"字幕文案要求"之后新增：

```markdown
### 动画指令要求（必需）

每段 `>` 口播之后，**必须**紧跟一行 `**动画**:` 描述该 step 画面上应发生的变化：

```markdown
> 口播文案...

**动画**: 画面变化描述
```

| 规则 | 说明 |
|------|------|
| 绑定 | 每段 `>` 后必须有 `**动画**:`，不可省略 |
| 内容 | 描述画面变化（什么入场/退场/高亮/移动），不写 CSS 细节 |
| 长度 | 一行，30-100 字 |
| 语义 | 动画内容必须图解对应的口播文案内容 |
| 色彩 | 使用品牌色语义词（陶土棕、赤陶红等） |

**编写步骤**：
1. 先写口播段——这一步在讲什么
2. 再想画面——如果观众只看画面，能不能理解口播在说什么
3. 写动画指令——这一步画面上具体发生什么变化

**动画指令 vs 画面描述 vs 视觉元素的区别**：
- `**画面描述**`：场景整体布局和氛围（宏观）
- `**视觉元素**`：场景中的数据素材和图表内容（原料）
- `**动画**`：每个 step 画面上的具体变化（微观、逐步）
```

常见错误表新增：

```markdown
| 错误 | 正确做法 |
|------|----------|
| 口播段后没有 `**动画**:` | 每段必须绑定动画指令，不可省略 |
| 动画指令和口播内容无关 | 动画必须图解口播——"说什么就展示什么" |
| 动画指令写 CSS/SVG 实现细节 | 只写"画面变化"，不写技术实现 |
| 动画指令过长（>100字） | 说明该拆段，一段只做一件事 |
```

### 6.4 `templates/standard-format.md` 模板更新

所有场景改为新格式。示例（场景1）：

```markdown
## 场景1：从一次熟悉的失控开始

**画面描述**: 左侧模拟终端界面，右侧是任务进度时间线...

**视觉元素**:

- → 第1步：任务拆解 ✅ 正常
  - → 第3步：代码修改 ⚠️ 小误判出现
    - → 第6步：修复方向偏移 ❌ 开始跑偏
      - → 第12步：结果全面偏离需求 💥 失控

> 你应该见过这种情况。Agent 一开始跑得很顺，计划像样，执行也快。前几步甚至让你觉得，这次可以完全放手了。

**动画**: 终端界面入场；时间线前3个节点依次点亮为陶土棕（正常态），执行进度平稳推进

> 但跑到第五步、第八步，事情开始变味。它修 bug 修错方向，而且越修越偏。

**动画**: 时间线节点从第5个开始变为暖金棕（警告），第8个变为赤陶红；终端中代码行标红闪烁

> 最后交出来的结果，已经不是你最初要的东西。你只能回滚，然后从头再来。

**动画**: 时间线末端节点爆裂为赤陶红碎片；"回滚"箭头从终点回弹到起点；整条时间线淡出
```

## 7. 向后兼容

| 场景 | 行为 |
|------|------|
| 旧格式 script.md（无 `**动画**:` 行） | `parse-markdown.js` 的 `animationIntents` 为空数组；`markdown-to-html` 回退到旧模式（自由编排） |
| 新格式 script.md 传入旧版 `parse-markdown.js` | `**动画**:` 行被忽略（不以 `>` 开头、不匹配已有规则）；功能正确但动画指令丢失 |
| 部分场景有动画指令、部分没有 | 有指令的场景按指令执行，没有的回退到自由编排 |

## 8. 验收标准

1. **`markdown-scriptwriter`** — 输出的 script.md 每段 `>` 后都有 `**动画**:` 行
2. **`parse-markdown.js`** — 正确解析 `**动画**:` 行，输出 `animationIntents[]` 数组，长度等于 `subtitles[]` 长度
3. **`markdown-to-html`** — step N 的动画内容与 `animationIntents[N]` 语义一致
4. **端到端验证** — 播放 final.mp4 时，口播在讲 X，画面就在展示 X 相关的动画
5. **向后兼容** — 旧格式 script.md 走完链路不报错（动画自由编排模式）

## 9. 实施顺序

1. **`parse-markdown.js`** — 新增 `**动画**:` 解析逻辑
2. **`markdown-scriptwriter/SKILL.md`** — 格式规范更新，新增动画指令规则
3. **`templates/standard-format.md`** — 模板全面更新为新格式
4. **`markdown-to-html/SKILL.md`** — 创作规则变更，强制按指令执行
5. **用现有 001 项目重新生成** — 验证端到端语义对齐

## 10. 一句话结论

**在 script.md 中为每段口播绑定显式的 `**动画**:` 指令，将"这段话配什么画面"的决策从 `markdown-to-html` 的自由创作前移到 `markdown-scriptwriter` 的有意识设计，消除口播与画面语义脱节的根因。**
